<figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*REJSTAfb5IiUjDaRm3YS5Q.png" /></figure><p>The Argo CD community is thrilled to announce Argo CD v3.3. This release delivers some long-awaited and highly anticipated features and improvements.</p><p>Let’s dive into what makes this release special.</p><h3><strong>PreDelete Hooks</strong></h3><p><a href="https://github.com/argoproj/argo-cd/pull/22288">PreDelete</a> Hooks has been a highly anticipated feature request in Argo CD. With v3.3, it’s finally here.</p><p>PreDelete hooks allow you to execute Jobs or other resources <strong>before</strong> Argo CD removes your application’s resources. Think of it as the missing piece in the application lifecycle puzzle — you’ve had PreSync, Sync, PostSync, and now you have PreDelete.</p><p>Until now, cleaning up before deletion required workarounds — external scripts, manual intervention, or hoping your finalizers would do the trick.</p><pre>apiVersion: batch/v1<br />kind: Job<br />metadata:<br />  name: cleanup-job<br />  annotations:<br />    argocd.argoproj.io/hook: PreDelete<br />spec:<br />  template:<br />    spec:<br />      containers:<br />      - name: cleanup<br />        image: alpine<br />        command: [&quot;sh&quot;, &quot;-c&quot;, &quot;echo 'Running pre-delete cleanup...'&quot;]<br />      restartPolicy: Never</pre><p>In the example we define a Job with the PreDeleteannotation. When the Application that owns this job is deleted, Argo CD first creates and runs this cleanup Job and waits for it to successfully complete. Only after the Job finishes does Argo CD proceed to delete the rest of the Application’s resources. If the Job fails the deletion is blocked.</p><p>Shoutout to <a href="https://github.com/pedro-cs-ribeiro">@pedro-cs-ribeiro</a> for making this happen!</p><h3>OIDC Background Token Refresh</h3><p>One of the most upvoted issues in Argo CD history has finally been resolved. Users integrating Argo CD with OIDC providers like Keycloak have long been frustrated by getting logged out after just 5 minutes — the typical access token lifespan even while actively working in the UI.</p><p>With this <a href="https://github.com/argoproj/argo-cd/pull/23727">PR</a> Argo CD now automatically refreshes OIDC tokens in the background before they expire. A new refreshTokenThreshold configuration option lets you specify when the server should proactively refresh tokens based on their remaining lifetime. This means no more unexpected logouts mid-workflow across Keycloak other OIDC providers.</p><p>Thanks to <a href="https://github.com/devopsjedi">@devopsjedi</a> for implementing this long-awaited quality-of-life improvement!</p><h3><strong>Source Hydrator Levels Up</strong></h3><p>The Source Hydrator has been one of the most actively developed features in Argo CD v3.x, and v3.3 brings a wave of improvements that make it more powerful and flexible than ever.</p><ul><li><a href="https://github.com/argoproj/argo-cd/pull/24277"><strong>Inline Parameter Support</strong></a><strong>: </strong>No more committing parameter files for every configuration change! With inline parameters, you can pass values directly during hydration. (by <a href="https://github.com/sangyeong01">@sangyeong01</a>)</li><li><a href="https://github.com/argoproj/argo-cd/pull/25056"><strong>Better Monorepo Support</strong></a><strong>: </strong>This brings significant improvements for monorepo workflows with source hydrator. (<a href="https://github.com/pbhatnagar-oss">by @pbhatnagar-oss)</a></li><li><a href="https://github.com/argoproj/argo-cd/pull/25150"><strong>Performance Improvements</strong></a><strong>: </strong>Source hydrator now avoids unnecessary repo-server calls, making the hydration process faster and more efficient. (by <a href="https://github.com/pbhatnagar-oss">@pbhatnagar-oss</a>)</li></ul><h3><strong>ClusterResourceWhitelist Now Supports Resource Names</strong></h3><p>One of the most upvoted RBAC <a href="https://github.com/argoproj/argo-cd/pull/24674">feature</a> requests has finally landed in v3.3.</p><p>Until now, clusterResourceWhitelist in AppProjects could only restrict access by API group and kind. If you allowed access to CustomResourceDefinitions, you allowed access to all CRDs. There was no way to say “this project can only manage these specific CRDs.”</p><p>With this feature, you can now limit cluster resource access to specific resource names:</p><pre>apiVersion: argoproj.io/v1alpha1<br />kind: AppProject<br />metadata:<br />  name: my-project<br />spec:<br />  clusterResourceWhitelist:<br />    - group: apiextensions.k8s.io<br />      kind: CustomResourceDefinition<br />      resourceNames:<br />        - crd1.example.com<br />        - crd2.example.com</pre><p>Shoutout to <a href="https://github.com/crenshaw-dev">@crenshaw-dev</a> for implementing this long-awaited feature!</p><h3><strong>Ability to use shallow clone for repositories</strong></h3><p>This <a href="https://github.com/argoproj/argo-cd/pull/24931">feature</a> allows Argo CD to fetch only the required commits instead of the entire repository history, reducing git fetch times from minutes to seconds for large repositories. Users can now add a depth flag when adding a repository to enable the shallow clone.</p><pre>argocd repo add &lt;repo-url&gt; --depth</pre><p>Thanks <a href="https://github.com/alexmt">@alexmt</a> for this long awaited feature</p><h3><strong>KEDA Gets First-Class Support</strong></h3><p>If you’re using KEDA (Kubernetes Event-driven Autoscaling), this release is for you.</p><ul><li><a href="https://github.com/argoproj/argo-cd/pull/25302"><strong>Pause Actions for ScaledObject and ScaledJo</strong>b</a>: You can now pause and resume KEDA resources directly from the Argo CD UI. This gives operators fine-grained control over autoscaling behavior — perfect for maintenance windows, debugging, or controlled rollouts.</li><li><a href="https://github.com/argoproj/argo-cd/pull/25106"><strong>ScaledJob Health Checks</strong></a><strong>: </strong>Argo CD now understands KEDA ScaledJob health status, so you’ll see accurate health indicators instead of generic “Unknown” states.</li></ul><p>Big thanks to <a href="https://github.com/rickbrouwer">@rickbrouwer</a> for KEDA support in Argo CD!</p><h3><strong>ApplicationSet Progressive Sync is now in Beta</strong></h3><p><a href="https://argo-cd.readthedocs.io/en/stable/operator-manual/applicationset/Progressive-Syncs/#use-cases"><strong>Progressive Sync</strong></a><strong> </strong>enables staged ApplicationSet updates by syncing Applications in defined groups and advancing only after health checks pass. This reduces blast radius when deploying or deleting changes across large fleets.<br />Thanks <a href="https://github.com/ranakan19">@ranakan19</a> for all your work on this feature!</p><h3><strong>Other Notable Changes</strong></h3><p>Here are some other improvements that made it into this release:</p><ul><li><a
